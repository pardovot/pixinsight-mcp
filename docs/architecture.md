# Architecture & Design

For the high-level pitch and quick start, see the README; this doc is the
component-level reference. The bridge wire format lives in
[bridge-protocol.md](bridge-protocol.md).

## Overview

A **stdio-based MCP server** (TypeScript) bridges AI assistants with a running
PixInsight instance. PixInsight has no socket/HTTP API, so all communication
goes through a **file-based command bridge**; commands are executed inside
PixInsight by a **native C++ module** hosting a non-blocking poller. The focus
is **post-processing** (after WBPP/pre-processing).

```
  Claude (Claude Code / Desktop)
        │  MCP (stdio, JSON-RPC 2.0)
        ▼
  MCP server — src/ (TypeScript → build/)
        │  ~/.pixinsight-mcp/bridge/  commands/<id>.json in, results/<id>.json out
        ▼
  MCPWatcher-pxm native module — module/  (pcl::Timer on PixInsight's event loop)
        │  MetaModule::EvaluateScript
        ▼
  Embedded JS handlers — generated from pjsr/pixinsight-mcp-watcher.js
        ▼
  PixInsight (stays interactive throughout)
```

## Components

### 1. MCP server (`src/`)

- Built with `@modelcontextprotocol/sdk`; spawned by the MCP host (Claude Code,
  Claude Desktop), speaks stdio JSON-RPC.
- Registers **14 tools** in three groups (see README for the table):
  image management (`open_image`, `save_image`, `close_image`,
  `list_open_images`, `get_image_statistics`), processing (`run_process`,
  `get_process_parameters`, `run_pixelmath`, `run_script`), and session
  (`get_history`, `undo`, `redo`, `snapshot`, `restore`).
- **Generic-runner design**: `run_process` runs *any* PixInsight process by
  class name; `get_process_parameters` introspects its settable parameters.
  There are deliberately no per-process tools (removed 2026-07-22) — every
  PixInsight process is `new X; set params; executeOn(view)`, so one generic
  tool covers all of them with zero per-process maintenance.
- `BridgeClient` (`src/bridge/client.ts`) owns the bridge protocol: atomic
  command writes, result polling with timeout, malformed-result fail-fast, and
  stale command/result cleanup at startup.

### 2. Native module (`module/`) — the runtime

- A compiled PCL module (`MCPWatcher-pxm.dll/.so/.dylib`) that owns a
  `pcl::Timer` firing on PixInsight's **own event loop** (default 300 ms).
  Because no script is running, PixInsight stays **fully interactive** while
  the bridge polls — you can pan/zoom/run processes and review the agent's
  work live. (A PJSR script cannot do this: PJSR is single-threaded, a running
  script holds the main thread, and a background `Timer` does not survive the
  script returning — verified.)
- **Thin shell**: every bridge command is delegated to the embedded JS handlers
  (`module/src/BridgeHandlersJS.h`, generated by `module/gen-handlers.mjs` from
  `pjsr/pixinsight-mcp-watcher.js`) via `MetaModule::EvaluateScript` on the
  root thread. Handler logic lives in ONE place — the JS.
- Small UI (Process > Utilities > MCP Watcher): version, status, Start/Stop;
  auto-starts when first opened.

### 3. JS watcher (`pjsr/pixinsight-mcp-watcher.js`) — dev fallback

- The same handler logic wrapped in a foreground polling loop, runnable from
  the Script Editor. Useful for iterating on handler logic without rebuilding
  the module, but it **freezes PixInsight's UI** while it runs — it is not the
  production channel.
- The section between the `__MCP_HANDLERS_BEGIN__`/`__MCP_HANDLERS_END__`
  sentinels is what gets embedded into the module; CI fails if the checked-in
  generated header drifts from it.

### 4. Bridge directory (`~/.pixinsight-mcp/bridge/`)

- `commands/` — MCP server writes, watcher reads + deletes.
- `results/` — watcher writes, MCP server reads + deletes.
- Atomic writes both ways (`<id>.tmp` → rename), 10-minute stale-command
  refusal on the watcher side, startup reaping on the server side. Details in
  [bridge-protocol.md](bridge-protocol.md).
- **Trust boundary**: any local process that can write to `commands/` can
  execute arbitrary code inside PixInsight — that is the bridge's job. Keep the
  directory user-private; do not point it at a shared or synced location.

### 5. Knowledge layer (`docs/workflows/`, skills)

- Per-acquisition-category playbooks (OSC-HOO, OSC-RGB, mono-RGB/LRGB/HaLRGB/
  SHO) that tell the agent *how* to process each data type; the
  `process-master` skill routes to the right playbook and drives the
  measure → configure → verify loop. This is data, not code — the tool surface
  stays generic.

## Delivery channels

1. **npm** — `@pardovot/pixinsight-mcp` (the MCP server).
2. **Signed PixInsight update repo** (`pi-repo/`) — ships the compiled native
   module via Resources > Updates; rebuilt with `npm run repo:build`. Currently
   signed with a **local** identity, so it validates only on machines where
   this PixInsight license is activated (public distribution needs CPD
   registration).
3. **Source build** (`module/`) — `build-pcl.mjs` → `build.mjs` → `sign.mjs` →
   `install.mjs`, all cross-platform Node scripts.

## Design decisions

- **File bridge, not sockets**: the only IPC PixInsight supports without core
  changes; latency (~poll interval) is irrelevant next to process runtimes.
- **Generic process runner, never per-process tools** — see CLAUDE.md; adding
  `run_bxt`-style wrappers is the anti-pattern this fork deliberately removed.
- **Measure → configure → verify**: tools and docs push the agent to measure
  the actual image before configuring a process, and to re-measure after
  (byte-identical stats = no-op = stop).
- **Handler logic in one place**: C++ is a non-blocking shell; behavior changes
  happen in the watcher JS and flow into the module at build time.
